# MAS-Lab — Harsh Code & Documentation Review

**Date:** 2026-06-29
**Scope:** `runtime/`, `ctl/`, `lab/` + `labs/` + `library-*`, `docs/` + tutorials
**Method:** static analysis, `git ls-files`/`git check-ignore`, `py_compile`, four parallel
component audits cross-checked against the announced design (README, CHANGELOG, blog).
Every CRITICAL/HIGH claim below was re-verified directly against the repo.

> **Tooling caveat.** The checked-in `.venv` is a macOS build (`.venv/bin/python` →
> `/opt/homebrew/.../python3.14`). It cannot run on Linux/CI, and the sandbox has only
> Python 3.10 while every package declares `requires-python = ">=3.11"`. As a result,
> **pytest / ruff / mypy could not be executed in this review.** Coverage, test pass/fail,
> and lint counts below come from static inspection only. *That the canonical dev
> environment is non-portable is itself a finding (H-ENV).* (The `.venv` is **not** tracked
> in git — good — but it is the only environment wired into the docs.)

---

## Verdict

The bones are real. `RuntimeKernel`/`RuntimeBuilder` (Mealy framing), `SessionController`,
and the `mas-ctl` CLI surface genuinely match the announced design — `mas-ctl` is in fact
one of the cleaner Python codebases here. The accusation that "vibe coding is everywhere"
is **half true**: the *core* `src` trees are largely free of TODO markers, bare excepts,
and commented-out code.

The rot is concentrated in three places: **packaging/release hygiene** (committed run
output that ships to PyPI, two wheels claiming the same files, four different version
numbers), the **lab/labs/component naming swamp**, and a **half-finished
Outshift→OSS cleanup** that the CHANGELOG declares "done" but the docs and tutorial
manifests contradict on 18 lines. Several CHANGELOG claims are simply false against the
code (hook plane "removed" — still present; terminology rename "complete" — 34 files
disagree). For a project whose entire pitch is *specification-driven, reproducible,
governable*, shipping non-deterministic wheels and a self-contradicting changelog is the
worst possible look.

Severity legend: **CRITICAL** = ships broken / data leak / non-deterministic ·
**HIGH** = misleads users or breaks the design story · **MEDIUM** = quality/consistency ·
**LOW** = polish.

## CRITICAL

### C1 — Local run output is committed *inside package source* and will ship in the wheel
`lab/src/mas/lab/output/` contains **10 tracked files**: seven `meta.json` under UUID run
dirs, a `logs/ui_feed.jsonl`, and `current_run_id.txt`. This sits under `src/mas`, which
`lab/pyproject.toml` packages — so stale local run artifacts (UUIDs, machine state) would
be **published to PyPI**. `.gitignore` has `**/output/*/` but the files were tracked
before the rule, so git ignores it.
*Fix:* `git rm -r --cached lab/src/mas/lab/output/`.

### C2 — Two wheels ship the same file path (non-deterministic packaging)
Both `mas-lab` (`lab/src/...`) and `mas-lab-bench` (`lab/components/bench/src/...`) install
`mas/lab/benchmark/runners/__init__.py` into the same PEP-420 namespace. `mas-lab` depends
on `mas-lab-bench`, so both are always installed together and one silently overwrites the
other. The umbrella copy is a 3-line stub. Install order decides behavior. **Verified:**
both paths present in `git ls-files`.

### C3 — Divergent duplicate module shipped by two wheels
`mas/lab/telemetry/otlp_push.py` is shipped by **both** `mas-lab` and `mas-lab-core`, and
the two copies **differ in content** (`diff -q` non-empty). Same collision class as C2 but
with active code drift — telemetry behavior depends on which wheel won the install race.

### C4 — Announced OSS cleanup never reached the docs/tutorials (18 hits)
CHANGELOG says "Open-source defaults use public endpoints (`standard:openai`,
`standard:ollama`)" and "Removed Outshift/Claris-specific infra bundles." The shipped
tutorials say otherwise on **18 lines**: every tutorial-02 agent manifest and several
tutorial-01 manifests hardcode `model: azure/gpt-4o*`; `docs/user-config.md:57` and
`docs/tutorials/00-environment-setup/README.md:104,306,312` describe the "Outshift proxy";
`docs/tutorials/03-experiments-and-analysis/README.md:826` references "Claris". A new OSS
user with a public OpenAI key follows the official Getting Started and hits unresolvable
`azure/`-prefixed model names. The flagship onboarding path is broken for the stated
audience.

## HIGH

### H-ENV — The canonical dev environment is non-portable
The only environment wired into the repo is a macOS `.venv` with host paths baked into
shebangs. It cannot run on Linux or CI. Contributors must reconstruct from `uv.lock`, but
no doc states this is mandatory. This blocked dynamic verification of the entire review.

### H1 — CHANGELOG claims contradicted by live code
- "**hook plane removed from OSS runtime**" — `runtime/src/mas/runtime/kernel/coord_hook.py`
  still defines `CoordHook` + `bind_coordination()` ("Optional M_coord hook on kernel hot
  path"), with 4 importers. `runtime/docs/production-path.md:45` even warns "do not
  reintroduce a parallel hook-plane." It was never removed.
- TODO.md: "**Terminology rename complete**" — the `governance`→`gov` rename is half done:
  `boundary/gov/` exists, but **34 files still say "governance"** in identifiers/docstrings
  (`GovernancePolicyEngine`, `ingress_governance_outcome`), and a dead
  `boundary/governance/budget.py` shim remains with **no importers**.

### H2 — The lab / labs / library-lab / "components" naming swamp
"lab" means four different things and "components" means two:
1. `lab/` = the `mas-lab` umbrella wheel (live CLI `mas.lab.cli:app`).
2. `lab/components/{core,bench,controller,content,tutorial}/` = five separately-published
   wheels.
3. `labs/` = the three paper lab *definitions* (`*.lab` dirs).
4. `library-lab/` = `mas-library-lab`, a 464-LOC facade that just re-wraps `library-eval`.
5. `lab/src/mas/lab/components/` = a *runtime module dir* (llm, mitm, ui…) unrelated to (2).

No newcomer can disambiguate `lab/components/core` (wheel) from
`lab/src/mas/lab/components/llm` (module) from `labs/design-space.lab` (data).

### H3 — Version numbers are incoherent (four values)
`VERSION` = `0.0.2`; top `pyproject.toml` = `0.0.1`; runtime/ctl/lab/library-standard/
library-samples/lab-core/lab-bench = `0.0.2`; **library-eval, library-lab, lab-content,
lab-controller, lab-tutorial = `0.1.0`**; blog brands "v0.1". A reader cannot tell what
version this is.

### H4 — CHANGELOG is structurally broken
Only `[Unreleased]` and `[0.1.0] — 2026-04-23` exist. **There is no `0.0.2` entry** even
though that is what `VERSION` ships. The ordering is inverted: `0.1.0` is dated *earlier*
(2026-04-23) than the "v0.1" blog post (2026-06-17), and is numerically *higher* than the
current `0.0.2`. README lists 4 packages; CHANGELOG lists `mas-lab-core`/`mas-lab-bench`;
`docs/packages-reference.md` enumerates **13** (incl. an otherwise-undefined
`mas-lab-controller`). Three package inventories, none agreeing.

### H5 — Canonical env var `MAS_DATA_ROOT` is documented nowhere
CHANGELOG: data paths are "`MAS_LABS_ROOT` / `MAS_DATA_ROOT` only." `MAS_DATA_ROOT` is real
in code (`lab/.../cli/commands/config.py`) but `grep MAS_DATA_ROOT docs/` = **0 hits**.
The tutorials instead document a *different, larger* set the CHANGELOG never blessed:
`MAS_TRACE_CACHE`, `MAS_DATA_CACHE`, `MAS_RUNS_ROOT`, `MAS_WORKSPACE_ROOT`.

## MEDIUM

### M1 — Dead directories (pure residue)
`runtime/src/mas/runtime/topology/`, `boundary/context/plugins/`, `boundary/grpc/` contain
**only** a stale `__pycache__` (sources deleted, darwin-3.14 `.pyc` left behind). Plus
`boundary/governance/` (dead shim). `runtime/tests/__pycache__/` still holds
`test_hook_plane.*.pyc` for a test source that no longer exists.

### M2 — Internally inconsistent plugin "contract"
`machines/design_pattern/protocol.py` defines `MealyPlugin` (Protocol), `DesignPatternPlugin`
(ABC), and alias `DesignPatternContract = MealyPlugin`. The kernel drives the loop via
`plugin.evaluate_next(...)` — but `evaluate_next` is **not in the Protocol**, only on the
ABC. For a "specification-driven" runtime, the spec disagrees with what the kernel calls.

### M3 — `print()` bleeding into importable library code
`mas-lab-bench` has **324** `print()` calls; `mas-lab-core` 28; runtime examples file 94
(see M5). A library that prints instead of logging is unusable in embedded contexts.

### M4 — Broad `except Exception` swallowing (counts)
runtime/src 18 · ctl/src 12 · lab/components 221 (bench 155). Worst silent swallows:
`runtime/.../registry.py:38`, `apps/_registry.py:26`, `library_catalog.py:20` (malformed
YAML → silently `{}`), `library_roots.py` (4× `return None`), `ctl/.../registry/catalog.py:46`.
`contracts/tool_contract.py:447` has a stripped `# Log warning?` TODO with `pass` left
behind — decision deleted, logging never implemented.

### M5 — Files flagged for deletion in TODO.md, partially undone
`runtime/examples/registry_manifest_demo.py` (carries all 94 runtime `print()`s) still
present. `fault_injection_plugin.py` and `interfaces/__init__.py` already deleted (2 of 3
done). The orphaned `lab/src/mas/lab/cli_demo.py` (365 lines) is imported nowhere yet its
sibling `cli/__init__.py:50` docstring still advertises a `demo` command that isn't wired.

### M6 — README documents 1 of 4 library packages
README "Packages" table lists only `mas-library-standard`; `mas-library-samples`,
`mas-library-eval`, `mas-library-lab` are real workspace members and dependencies but
undocumented.

### M7 — `chat_cmd` is a god command
`ctl/src/mas/ctl/cli/commands/chat.py:78-272`: 27 params, ~85 lines of decorators, ~160-line
body (manifest load, dotenv, config merge, infra resolve, observability, run loop). `tui_cmd`
duplicates ~40 lines of the same merge/resolve logic.

### M8 — Empty/stub scaffolding & untested package
`lab/src/mas/lab/benchmark/pipeline/steps/` is `__pycache__`-only; `mas-lab-content`
(856 LOC) has **0 tests**.

## LOW

- **L1** Duplicate/split imports: `chat.py:14&17` and `runtime_cli.py:96-97` both import the
  controller twice; `validate.py` imports `yaml` at module top *and* inside the function;
  `run_mas.py:63` does a function-local `import yaml`.
- **L2** Type-hint gaps on CLI surface: `chat_cmd`, `tui_cmd`, the `observability_options`
  decorator, `_print_result` are unannotated; `RuntimeKernel.snapshot()/restore()` use bare

- **L4** Test scaffolding shipped in the wheel: `ctl/src/mas/ctl/testing/` is packaged but
  used only by repo tests.
- **L5** Broken doc refs in source docstrings: `kernel/coupling.py:9` → missing

- **L7** Duplicated evaluate-loop in `kernel/orchestrator.py` (inlined 2×, helper used once).
- **L8** `doc-map.md` manually mirrors the entire `mkdocs.yml` nav — a second source of

## What is actually good (for balance)

- `RuntimeKernel` + `RuntimeBuilder` are real, coherent, properly exported; the Mealy
  framing matches the announced design. No `AgentBuilder`/`AgentRuntime` left in `src`.
- `mas-ctl`: 0 bare excepts, 0 TODOs, 0 commented-out blocks, 1 `print()`. Every announced
  command (`chat`, `run-mas`, `validate`, `compose`) exists and is registered.
  `SessionController` is genuinely the shared session abstraction it claims to be.
- `site/` is **not** committed (gitignored). Stale terminology is confined to the CHANGELOG
  removal notes — it has *not* leaked into the published docs.
- Governance-as-plugin-list, the `lab-config.yaml` rename, and the removal of
  `MAS_BENCHMARK_ROOT`/`MAS_LAB_OUTPUT` are all correctly reflected in docs and code.
- Python-version story (3.11+ everywhere, CI 3.11+3.13) is consistent.

## Decisions (2026-06-29)

This repo is being prepared as a **fresh, single-commit, first public release at `v0.1.0`**.
That reframes several findings:

- **Version:** everything converges to **`0.1.0`** (not `0.0.2`). H3 becomes a simple
  one-pass alignment.
- **CHANGELOG:** collapses to a single **"Initial release — v0.1.0"** entry. Because there
  is no prior *public* version, the changelog cannot (and will not) claim anything was
  "removed/migrated/changed." This **dissolves H1 and H4**: the hook-plane "removed" claim
  and the "terminology rename complete" claim simply disappear from history. The
  `governance`→`gov` inconsistency (M1/H1) is still worth fixing for a clean first release,
  but it is no longer a *contradiction with the changelog* — just internal tidiness.
- **TODO.md** ("TO REMOVE…", "rename complete") is dev scratch and must not appear in a
  first public release — delete it (its 3 actions are folded into the roadmap).
- **Git history:** all of this lands as a single amended initial commit, so per-commit
  hygiene is irrelevant; only the final tree state matters.

- **Deferred (test-gated):** **R6.1** — the physical rename of the `mas.lab.components`
  runtime-module package. It is referenced by *dynamic string imports* (subprocess
  `python -m …` backends, a deepeval class path) that static analysis cannot verify;
  renaming blind risks a runtime break in the controller's metrics/llm_mock backends.
  The exact, exhaustive rename recipe is recorded in
  [ADR 0001](references/adr-0001-lab-terminology.md) to be applied in a change set
  where `pytest` + a backend-spawn smoke test can confirm it.
- **Reassessed (not a defect):** the "324 `print()` in bench" finding — those are
  overwhelmingly legitimate CLI/report output, not debug residue; converting them to
  logging would break the CLI UX and stdout-capturing tests. Left as-is.

## Verify on a developer machine (commands)

The authoring environment had no Python ≥3.11 and no network, so the suite was not
run here. From the repo root on a dev machine:

```bash
uv sync                              # build the env from uv.lock (NOT the macOS .venv)
uv run ruff check .                  # lint
uv run mypy runtime/src ctl/src      # types
uv run pytest -q                     # full suite (incl. new content + packaging-hygiene tests)
uv build                             # build all wheels
# Confirm no two wheels ship the same file path, and the umbrella no longer ships
# mas/lab/benchmark or mas/lab/telemetry.
```

If anything fails, the highest-risk areas to check first are the `chat`/`tui`
infra-resolution helper (`mas.ctl.session.infra_resolve`) and the telemetry owner
move (`mas-lab-core` now owns `mas/lab/telemetry/otlp_push.py`).
