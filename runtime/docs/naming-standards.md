# Agent Identity and Naming Standards

## URN Standard Scheme

To ensure observability, security, and unique addressability across distributed MAS, all entities MUST be identified by a URN following this scheme:

```text
urn:mas:{realm}:{class}:{type}:{name}:{instance}
```

### Components

1. **Realm**: Organizational boundary (e.g., `prod`, `dev`, `corp`).
2. **Class**: High-level category:
    * `agent`: Autonomous actor.
    * `tool`: Passive capability.
    * `resource`: Shared data/storage.
    * `service`: Infrastructure component.
3. **Type**: Specific class/role definition (e.g., `sre-lead`, `postgres-db`).
4. **Name**: Human-readable logical name (e.g., `primary`, `checkout-bot`).
5. **Instance** (Optional): Unique UUIDv4 for a specific runtime execution.

### Examples

* **Logic Role**: `urn:mas:prod:agent:sre:lead` (The broad concept of the SRE Lead).
* **Specific Agent**: `urn:mas:prod:agent:sre:lead:01` (A specific deployed agent).
* **Runtime Instance**: `urn:mas:prod:agent:sre:lead:01:c29dfb02-318e-4ca8` (This exact process).
* **Tool**: `urn:mas:global:tool:api:github:v3`.
* **Resource**: `urn:mas:prod:resource:db:users:primary`.

## Identity Lifecycle

1. **Allocation**:
    * **Static IDs** (Role/Name) are defined in the agent manifest or MAS topology configuration.
    * **Instance IDs** are auto-generated by the runtime at agent startup.

2. **Resolution**:
    * Agents address each other by **Logical ID** (`urn:mas:prod:agent:sre:lead`).
    * The `CommunicationPlane` (agent-remote Plugin) resolves Logical ID -> Instance ID(s) via the Service Registry.

3. **Governance**:
    * Policies are usually bound to **Type** or **Name** (e.g., "Allow `agent:sre:*` to call `tool:github:*`").
    * Audit logs record **Instance IDs** for forensics.

## Task-Based Access Control (TBAC)

Task-scoped governance (TBAC) and execution sandboxing are **not** part of the
OSS release. See `mas-lab-internal` (`docs/contracts/tbac-contract.md`,
`sandbox-contract.md`).
